# Review: feature/ilogger-rework

## Plan Compliance
| Finding | Severity | Why It Matters |
|---------|----------|----------------|
| DPanic method specified in plan Phase 3 (ILogger dispatches at Error level, throws in DEBUG builds) is not implemented. No `DPanic` method exists in `ILogger.cs`. | Medium | Missing API surface from the plan. Users expecting DPanic for dev-time fail-fast patterns will not find it. Should be added in a later phase or the plan amended. |
| Plan Phase 1 specifies deleting `LogScope.cs` (the AsyncLocal version), but Phase 3 recreates it as a struct-based ILogger+IDisposable. Implementation correctly follows this: the old LogScope was removed and a new struct-based one was created. However, plan Phase 1 also says to delete `StructuredPathEmitter.cs`, which was deleted, but the plan says to "refactor" it first to emit inline JSON. Instead, the inline JSON logic was added directly to `MethodEmitter.cs`. | Low | Minor deviation from plan wording but the intent (inline JSON in method body) is fully achieved. No functional gap. |
| Plan Phase 7 specifies `DependencyInjection/LogsmithServiceCollectionExtensions.cs` under `src/Logsmith/`. Implementation places it at `src/Logsmith.Extensions.Logging/LogsmithServiceCollectionExtensions.cs`. | Low | Different location than planned, but arguably better: DI integration belongs in the Extensions.Logging project which already has the Microsoft.Extensions.DependencyInjection dependency. The plan's location would have required adding DI packages to the core Logsmith library. |
| Plan Phase 6 specifies `LSMITH013` diagnostic for broken chains. The descriptor is defined in `DiagnosticDescriptors.cs`, but the `ChainAnalyzer` never actually emits this diagnostic -- it silently stops walking the chain. | Medium | The diagnostic was planned to warn users when they break a chain by storing intermediates in a variable. Without it, users get silent fallback to default interface methods with no optimization, and no indication that their chain is not being intercepted. |
| Plan Phase 2 specifies `LoggerContext.CreateChild` "creates child context with parent link, shared sinks" and mentions "independent level override" on child contexts. The implementation has no per-context minimum level override -- all level checks delegate to `LogManager.IsEnabled(level, category)`. | Low | The plan mentions minimum level override capability on child contexts, which is not implemented. This is minor because category-based overrides via LogManager serve a similar purpose. |
| Plan Phase 4 specifies buffer renting from a "ThreadBuffer pool" in `LogHandlerCore`. The implementation uses `new ArrayBufferWriter<byte>()` directly, allocating on every enabled log call. | Medium | The plan explicitly calls for pooled buffers to achieve zero-allocation goals. Current implementation allocates two `ArrayBufferWriter<byte>` instances per log call (text + JSON). For a library targeting zero allocation, this is a notable deviation that should be addressed before release benchmarking. |
| Phases 6 and 7 are implemented (interceptors, DI, MEL bridge) despite the workflow.md showing `phases-complete: 5` at the time of suspend. The latest commit updates workflow to show phase REVIEW with all 7 complete. | Low | The workflow metadata was updated. Implementation is ahead of what the suspend state suggested, which is fine -- just noting the session 3 work completed phases 6-7. |

## Correctness
| Finding | Severity | Why It Matters |
|---------|----------|----------------|
| `LoggerContext.GetCachedPathBytes()` has a race condition. `_cachedPathVersion` and `_cachedPathBytes` are read/written without synchronization. Two concurrent calls could both see a stale version, both compute new bytes, and one could write `_cachedPathBytes` while another reads it mid-flight. The fields are not volatile and there is no lock. | Medium | In practice, worst case is redundant path computation (benign) since the `byte[]` reference write is atomic on .NET. However, the version check + bytes write is not atomic: a thread could read `_cachedPathVersion == currentVersion` (set by another thread) but get an old `_cachedPathBytes`. Adding `Volatile.Write`/`Read` on both fields or using a simple lock would close this gap. |
| `PathNode.WriteUtf8Path` allocates a `string[]` array on every call via `var segments = new string[depth]`. For a library focused on zero allocation, this is a hot-path allocation during every dispatch through a context with path nodes. | Medium | Path rendering happens on every log dispatch for contexts with path nodes. The allocation could be avoided with stackalloc of `nint` pointers or a fixed-size buffer for shallow depths (most real-world paths are <8 deep). |
| `DefaultLogFormatter.FormatPrefix` requests a 512-byte span via `output.GetSpan(512)` and writes into it without bounds checking. If category name + path + tag exceed ~480 bytes (accounting for timestamp/level prefix), it will write past the span boundary, causing an `IndexOutOfRangeException`. | High | Long category names, deep paths, or long tags could crash the formatter. The 512-byte estimate is generous for typical use, but there is no fallback or dynamic growth. A bounds check or computed size should replace the fixed constant. |
| `LogsmithLogger.Log<TState>` in the MEL bridge uses `stackalloc byte[Math.Min(message.Length * 3, 4096)]`. If the UTF-8 encoded message exceeds 4096 bytes, the `Utf8.FromUtf16` call silently truncates -- `bytesWritten` will be less than the full message, and only the truncated portion is logged. The `OperationStatus` return value is not checked. | Medium | Log messages longer than ~1365 characters (worst case UTF-8) will be silently truncated. The method should check the status and fall back to a heap allocation for the overflow case. |
| `NullLogger` creates a `LoggerContext("Null")` as its sentinel context. If someone calls `NullLogger.Instance.Context.Dispatch(...)`, the dispatch will actually execute and try to write to sinks (if LogManager is initialized). The NullLogger's `IsEnabled` override returns false, but direct `Context.Dispatch` bypasses that check. | Low | NullLogger's contract is that all operations are no-ops. Direct `Context.Dispatch` access would violate that, but in practice callers use the ILogger interface methods which check `IsEnabled` first. The `Context` property is required by the interface, so this is a known trade-off. |
| `LogHandlerCore` does not dispose the `Utf8JsonWriter` or `ArrayBufferWriter` instances. The `Utf8JsonWriter` implements `IDisposable` and holds internal buffers. | Low | For a ref struct that lives on the stack and is used within a single method, the GC will reclaim these promptly. However, `Utf8JsonWriter` may hold a rented array internally. Not disposing means the rented array is reclaimed by GC finalization rather than being returned to the pool promptly. Over time this could increase memory pressure under high-throughput logging. |
| `TimingOperation` and `LogScope` are value types (structs) implementing `IDisposable`. If used without `using`, or if copied (e.g., passed by value to another method), the dispose behavior may be surprising -- one copy's `_completed`/`_disposed` flag won't affect the other's. | Low | This is a known C# pattern trade-off. The types are documented to be used via `using`, and the plan acknowledges that struct types require cast to `ILogger` for default method access (boxing). The struct semantics are appropriate for the use case. |
| Rate limiting in interceptor-generated code (`InterceptorEmitter.EmitChainStartInterceptor` for `RateLimited`) has a TOCTOU race: the window check and count reset are not atomic. Two threads could both see `__now != __window`, both reset the counter, and both increment -- allowing more messages than `maxPerSecond` in a burst. | Low | Rate limiting is inherently approximate in concurrent scenarios. The implementation provides "best effort" rate limiting, which is typical for logging libraries. The race window is small and the consequence (a few extra log messages) is benign. |

## Security
| Finding | Severity | Why It Matters |
|---------|----------|----------------|
| Interceptor-generated code embeds the full file path of the caller in the generated source: `CallerFile = @"C:\Users\dev\src\MyApp\Service.cs"`. This path is baked into the compiled assembly as a string literal. | Low | Full file paths in log output can leak developer machine directory structures and usernames. This is identical to how `[CallerFilePath]` works in standard .NET and is expected behavior. Consumers should be aware that file paths appear in log output and in the compiled binary. |
| `CallerArgumentExpression` captures the source code expression text as JSON property names. Expressions like `secretPassword` or `apiKey` would become JSON property names in structured logs. | Low | This is by design (structured logging captures variable names). Users should be aware that variable names become property names in structured log output. This is the same behavior as Serilog's `{@PropertyName}` and MEL's `{PropertyName}` patterns. |

## Test Quality
| Finding | Severity | Why It Matters |
|---------|----------|----------------|
| No tests for `LogHandlerCore` disposal or buffer cleanup. If `ArrayBufferWriter` or `Utf8JsonWriter` hold rented arrays, there are no tests verifying they are returned. | Low | Since the handlers are ref structs on the stack, this is less critical, but a test verifying no memory leaks under sustained load would increase confidence. |
| Chain method tests (`Sampled`, `RateLimited`, `Tagged`) only verify the default stub behavior (return `this`). There are no runtime tests verifying that interceptor-generated chain methods actually perform sampling, rate-limiting, or tag propagation when interceptors are active. | Medium | The interceptor tests verify code generation output (string matching on generated source), but there are no end-to-end tests that compile+run the generated interceptors and verify sampling actually drops messages or rate-limiting caps throughput. Integration tests exist for `When` and `Tagged` but not for `Sampled` or `RateLimited`. |
| `RecordingSink` is not thread-safe (`List<CapturedEntry>` is not concurrent), but it is used in tests that may involve multi-threaded dispatch. No test explicitly validates thread-safety. | Low | For test purposes, most tests run single-threaded. However, if any test uses async or concurrent dispatch, entries could be lost or corrupted. A `ConcurrentBag` or lock would be safer. |
| No negative tests for `PathNode.WriteUtf8Path` with an undersized destination buffer. If the span is too small, `Encoding.UTF8.GetBytes` will throw. | Low | The code calls `CalculateMaxByteCount()` first and allocates that size, so in practice this is safe. However, a test confirming the calculation is correct for multi-byte UTF-8 characters would increase confidence. |
| No tests for `LoggerContext.Dispatch` behavior when a sink throws. The `LogManager.Dispatch` catches exceptions and calls the error handler, but tests don't verify that dispatch continues to subsequent sinks after one throws. | Medium | If a sink throws, the error handler is invoked, but the test suite does not verify that remaining sinks still receive the log entry. This is important for production reliability -- a single failing sink should not block others. |
| `DiIntegrationTests` does not test the `AddLogsmith` method being called when LogManager is already initialized (double-init scenario). | Low | `AddLogsmith` calls `LogManager.Initialize` which throws on double-init. There is no test verifying the error message or suggesting the use of Reconfigure. |
| No tests verify that the `Log` static class `[Conditional]` methods actually strip the call site when the defining symbol is absent. The plan calls for this explicitly. | Medium | The `[Conditional]` stripping is a compile-time behavior that requires compiling test code with and without the symbols defined, which is tricky in a test project. However, the prototype `ConditionalTest` project validates this manually. An automated test (possibly a generator test checking for or against the presence of calls) would be more robust. |

## Codebase Consistency
| Finding | Severity | Why It Matters |
|---------|----------|----------------|
| `ILogger.cs` has six nearly-identical `DispatchHandler` private methods (one per handler type). The plan mentions a shared `LogHandlerCore` to avoid this duplication, and indeed the handlers delegate to it, but the `ILogger` interface still has six copy-pasted dispatch helpers that differ only in the handler type parameter. | Medium | This is a consequence of C# not allowing generic ref struct parameters in interface default methods. The duplication is functionally necessary but could be reduced with a code generator or T4 template. As-is, any bug fix must be applied to all six methods. |
| The Logsmith `ILogger` naming conflicts with `Microsoft.Extensions.Logging.ILogger`. The resolution is documented (using aliases), and benchmark/bridge code correctly qualifies types. This is consistent with the pattern used by Serilog (`Serilog.ILogger`). | Low | The naming conflict is intentional and documented. Samples show the alias pattern. This is a known trade-off that was explicitly decided. |
| `LogScope` and `TimingOperation` use `string.Format`-style string interpolation for their log messages (`$"Operation '{_operationName}' completed..."`) rather than using the handler-based ILogger API. This means their timing messages lack structured JSON output. | Medium | These types bypass the structured logging path by using `Encoding.UTF8.GetBytes(message)` directly. Their messages won't have JSON-structured properties (e.g., elapsed time as a numeric property). This is inconsistent with the dual-buffer philosophy. The timing/scope messages are "meta" messages, so this may be acceptable, but it should be a conscious decision. |
| The `Log` static class has exception overloads for `Error` and `Critical` but not for `Trace`, `Debug`, `Information`, or `Warning`. The `ILogger` interface has exception overloads for all six levels. | Low | Asymmetric API surface. While `Error` and `Critical` are the most common levels for exception logging, users may want to log exceptions at other levels (e.g., `Debug` with a caught/retried exception). |
| `SinkSet` in the diff simplified from having `TextSinks`/`StructuredSinks` split to a single `Sinks` array. The `AllSinks` property referenced in the plan was replaced by `Sinks` directly. `DisposeSinksAsync` is on `SinkSet`, while `DisposeAllAsync` is on `LogConfig`. This is internally consistent. | Low | Clean simplification that follows the plan's intent. |
| Existing test files consistently use `[SetUp]/[TearDown]` with `LogManager.Reset()`, and all new test files follow this pattern. New tests use NUnit `Assert.That` with constraint-model assertions, matching existing style. | Low | Good consistency. No concerns. |

## Integration / Breaking Changes
| Finding | Severity | Why It Matters |
|---------|----------|----------------|
| `ILogSink.Write` signature changed from `Write(in LogEntry entry, ReadOnlySpan<byte> utf8Message)` to `Write(in DispatchInfo info)`. This is a **breaking change** for all existing sink implementations. | High | Any third-party or user-defined sink must update its `Write` method signature and adapt to the new `DispatchInfo` fields. The `LogEntry` type no longer exists. Migration requires updating the parameter type and accessing message bytes via `info.Utf8Message` instead of the separate parameter. |
| `ILogFormatter` signature changed from using `LogEntry` to `DispatchInfo`. `FormatPrefix(in DispatchInfo info, ...)` and `FormatSuffix(in DispatchInfo info, ...)`. | High | Any custom formatter must be updated. The fields are similar but the type is different (`ref struct` vs `record struct`). |
| `LogEntry` type removed entirely. | High | Direct references to `LogEntry` in user code will fail to compile. This is the intended replacement (DispatchInfo), but any code constructing `LogEntry` must be migrated. |
| `IStructuredLogSink` removed, `WriteProperties` removed. | High | Any sink implementing `IStructuredLogSink` must switch to the unified `ILogSink.Write(in DispatchInfo)` pattern. Structured data is now available via `info.Utf8Json` as pre-built UTF-8 bytes. |
| `LogScope` (AsyncLocal-based) removed and replaced with struct-based version. | High | Any code using the old `LogScope.Push()` / `LogScope.Current` ambient scoping will break. The replacement is `logger.Scoped("segment")` which is explicit and struct-based. There is no AsyncLocal-based ambient scoping anymore. |
| `SinkSet` no longer has `TextSinks`/`StructuredSinks` split. Internal type, but if anyone accessed it via reflection or internal APIs, this breaks. | Low | Internal type -- unlikely to affect consumers. |
| New `Logsmith.ILogger` interface name conflicts with `Microsoft.Extensions.Logging.ILogger`. Projects referencing both must use aliases or full qualification. | High | Any project that imports both `Logsmith` and `Microsoft.Extensions.Logging` namespaces will get ambiguous reference errors for `ILogger`. The resolution is well-documented (aliases, explicit qualification), but this will affect every existing project that upgrades and uses both namespaces. |
| `LogManager.GetLogger()` and `LogManager.GetLogger<T>()` are new APIs returning `Logsmith.ILogger`. These are additive, not breaking. | Low | New convenience APIs. No breaking change. |
| New `InterceptorsNamespaces` MSBuild property added to `Logsmith.props`. Consumers using interceptors need this property. It's automatically set via the props file, so no manual action needed. | Low | Additive change. The props files are imported automatically by NuGet. |
| New dependency: `Logsmith.Extensions.Logging` project adds `Microsoft.Extensions.DependencyInjection.Abstractions` package reference for `AddLogsmith` extension method. | Low | New dependency on the Extensions.Logging project only (not the core library). This is expected for DI integration. |
| `BufferedLogSink` internal `BufferedEntry` changed from a simpler structure to include all `DispatchInfo` fields plus a combined `Buffer` with offsets for message/json/path. | Medium | Any custom sink extending `BufferedLogSink` and overriding `WriteBufferedAsync` must update to the new `BufferedEntry` shape. The `ToDispatchInfo()` helper makes reconstruction easy. |
| `RecordingSink.CapturedEntry` record gained new fields: `Tag`, `CallerFile`, `CallerLine`, `CallerMember`, `ThreadId`, `ThreadName`, `JsonMessage`, `Path`. | Medium | Test code using `CapturedEntry` positional constructor will need to be updated for the new fields. Existing tests were already updated in the diff. |
